# TDX Example

A complete example demonstrating mosaik's Intel TDX (Trust Domain Extensions) support. This example builds a mosaik application that uses hardware attestation to create a secure data stream and a replicated map — only peers running inside a TDX enclave with the correct firmware measurements can participate.

## Overview

The example compiles into a single binary that runs in two modes:

- **Producer mode** (outside TEE) — publishes messages on a stream that requires TDX attestation from consumers
- **Consumer mode** (inside TDX) — generates a hardware attestation ticket, publishes it via discovery, subscribes to the secure stream, and writes received messages into a replicated `SecureObservers` map shared only with peers that have identical TDX measurements

The mode is selected automatically based on whether TDX hardware is detected at runtime.

## Features Demonstrated

- **TDX attestation tickets** — generating and installing hardware-signed attestation via `network.tdx().install_own_ticket()`
- **TDX measurement validation** — using `Tdx::new().require_mrtd(...)` to enforce that consumers run the expected firmware
- **Local measurement matching** — using `Tdx::new().require_own_mrtd()` and `.require_own_rtmr2()` to require peers to have the same measurements as the local machine, without hardcoding measurement values
- **`declare::stream!` macro with `require_ticket`** — compile-time stream declaration with baked-in TDX validation
- **`declare::collection!` macro with `require_ticket`** — compile-time collection declaration with TDX validation derived from local measurements
- **Build-time image creation** — the `build.rs` uses `mosaik::tdx::build::alpine()` to cross-compile the binary for TDX and package it with an Alpine Linux guest image

## Architecture

```
┌───────────────────────┐          ┌──────────────────────────────────────┐
│   Producer (host)     │          │  Consumer (TDX guest)                │
│                       │          │                                      │
│  SecureStream         │◄─────────│  TDX Quote + ExtraData               │
│  .producer()          │  gossip  │  .install_own_ticket()               │
│                       │          │                                      │
│  sends messages ──────┼──────────┼─► receives messages                  │
│  (if consumer valid)  │  QUIC    │  (if MR_TD matches)                  │
│                       │          │                                      │
│                       │          │  SecureObservers Map (Raft group)    │
│                       │          │  ← only peers with same MRTD+RTMR2 → │
│                       │          │  writes (peer_id, last_message)      │
└───────────────────────┘          └──────────────────────────────────────┘
```

## Prerequisites

- Rust toolchain with `x86_64-unknown-linux-musl` target installed
- For running the consumer: a TDX-capable host with QEMU and TDX support enabled
- The build process automatically downloads Alpine Linux, a TDX kernel, and OVMF firmware

## Building

```bash
cargo clean && cargo build -p tdx-example --release
```

The build script will:

1. Generate a unique build ID (shared between host and guest binaries)
2. Cross-compile the binary for `x86_64-unknown-linux-musl`
3. Download and package Alpine Linux minirootfs, TDX kernel, and OVMF firmware
4. Create a self-extracting launch script for the TDX guest
5. Pre-compute the `MR_TD` measurement for the guest image
6. Create a self-contained deployable image that can be launched on TDX host

## Running

Ensure that you're running binaries produced by the same build, because the network id is generated randomly per build. All peers sharing the same network id will automatically discover each other through mosaik's built-in DHT discovery — no manual peer configuration or address exchange is needed.

**Producer (on any non-tdx machine):**

```bash
./target/release/tdx-example
```

If TDX is not available, the binary runs in producer mode — it creates a `SecureConsumer` stream and publishes messages every 2 seconds.

**Consumer (inside TDX guest):**

Upload the tdx artifacts generated by the alpine tdx builder to you tdx host:

```bash
scp -r target/release/tdx-artifacts/tdx-example/alpine/ ubuntu@tdx-host:~/tdx-example
```

Launch two instances of the the generated TDX guest image on a TDX-capable host. 

```bash
sudo ./tdx-example/tdx-example-run-qemu.sh
```

The guest binary detects TDX hardware, generates an attestation ticket, and subscribes to the stream. The producer validates the consumer's `MR_TD` measurement before accepting the subscription. Since all instances share the same network id, they self-discover automatically via DHT — the producer and consumers will find each other without any explicit peer addresses.

## How the Stream Declaration Works

```rust,ignore
declare::stream!(
    pub SecureConsumer = String,
    "mosaik.examples.tee.tdx.SecureConsumer",
      consumer require_ticket: Tdx::new()
          .require_mrtd("91eb2b44..."),
);
```

This declares a stream where:
- The datum type is `String`
- Consumers must present a valid TDX ticket (the `consumer require_ticket` key applies validation only on the producer side, validating incoming consumers)
- The `MR_TD` measurement in the consumer's TDX Quote must match the specified value — this value is determined at build time by the image builder

## How the Collection Declaration Works

```rust,ignore
declare::collection!(
    pub SecureObservers = mosaik::collections::Map<PeerId, String>,
    "mosaik.examples.tee.tdx.SecureObservers",
      require_ticket: Tdx::new()
        .require_own_mrtd().expect("TDX support")
        .require_own_rtmr2().expect("TDX support")
);
```

This declares a replicated `Map` collection where:

- All group members must present a valid TDX ticket
- The `MR_TD` and `RTMR2` measurements in each member's TDX Quote must match the local machine's own measurements — the values are read from TDX hardware at runtime via `require_own_mrtd()` and `require_own_rtmr2()`, so there is no need to hardcode measurement hex strings
- Only TDX peers running the exact same firmware and OS configuration can join the consensus group backing this collection

In the consumer flow, each received stream message is inserted into this map as `(peer_id, message)`, and a background task watches for updates and prints the current map contents. This demonstrates how a TEE-gated collection can serve as shared state exclusively among attested peers.
