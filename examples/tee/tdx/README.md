# TDX Example

A complete example demonstrating mosaik's Intel TDX (Trust Domain Extensions) support. This example builds a mosaik application that uses hardware attestation to create a secure data stream — only consumers running inside a TDX enclave with the correct firmware measurement can receive data.

## Overview

The example compiles into a single binary that runs in two modes:

- **Producer mode** (outside TEE) — publishes messages on a stream that requires TDX attestation from consumers
- **Consumer mode** (inside TDX) — generates a hardware attestation ticket, publishes it via discovery, and subscribes to the secure stream

The mode is selected automatically based on whether TDX hardware is detected at runtime.

## Features Demonstrated

- **TDX attestation tickets** — generating and installing hardware-signed attestation via `network.tdx().install_own_ticket()`
- **TDX measurement validation** — using `TdxValidator::new().require_mrtd(...)` to enforce that consumers run the expected firmware
- **`stream!` macro with `require_ticket`** — compile-time stream declaration with baked-in TDX validation
- **Build-time image creation** — the `build.rs` uses `mosaik::tdx::build::alpine()` to cross-compile the binary for TDX and package it with an Alpine Linux guest image

## Architecture

```
┌───────────────────────┐          ┌──────────────────────────┐
│   Producer (host)     │          │  Consumer (TDX guest)    │
│                       │          │                          │
│  SecureStream         │◄─────────│  TDX Quote + ExtraData   │
│  .producer()          │  gossip  │  .install_own_ticket()   │
│                       │          │                          │
│  sends messages ──────┼──────────┼─► receives messages      │
│  (if consumer valid)  │  QUIC    │  (if MR_TD matches)      │
└───────────────────────┘          └──────────────────────────┘
```

## Prerequisites

- Rust toolchain with `x86_64-unknown-linux-musl` target installed
- For running the consumer: a TDX-capable host with QEMU and TDX support enabled
- The build process automatically downloads Alpine Linux, a TDX kernel, and OVMF firmware

## Building

```bash
cargo build -p tdx-example --release
```

The build script will:

1. Generate a unique build ID (shared between host and guest binaries)
2. Cross-compile the binary for `x86_64-unknown-linux-musl`
3. Download and package Alpine Linux minirootfs, TDX kernel, and OVMF firmware
4. Create a self-extracting launch script for the TDX guest
5. Pre-compute the `MR_TD` measurement for the guest image
6. Create a self-contained deployable image that can be launched on TDX host

## Running

Ensure that you're running binaries produced by the same build, because the network id is generated randomly per build

**Producer (on any machine):**

```bash
./target/release/tdx-example
```

If TDX is not available, the binary runs in producer mode — it creates a `SecureConsumer` stream and publishes messages every 2 seconds.

**Consumer (inside TDX guest):**

Upload the tdx artifacts generated by the alpine tdx builder to you tdx host:

```bash
scp -r target/release/tdx-artifacts/tdx-example/alpine/ ubuntu@tdx-host:~/tdx-example
```

Launch the generated TDX guest image on a TDX-capable host. 

```bash
sudo ./tdx-example/tdx-example-run-qemu.sh
```

The guest binary detects TDX hardware, generates an attestation ticket, and subscribes to the stream. The producer validates the consumer's `MR_TD` measurement before accepting the subscription.

## How the Stream Declaration Works

```rust,ignore
declare::stream!(
    pub SecureConsumer = String,
    "mosaik.examples.tee.tdx.SecureConsumer",
    consumer require_ticket: TdxValidator::new()
        .require_mrtd("91eb2b44..."),
);
```

This declares a stream where:
- The datum type is `String`
- Consumers must present a valid TDX ticket (the `consumer require_ticket` key applies validation only on the producer side, validating incoming consumers)
- The `MR_TD` measurement in the consumer's TDX Quote must match the specified value — this value is determined at build time by the image builder

This applies to collections and groups as well.
